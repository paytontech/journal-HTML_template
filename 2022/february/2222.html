Lucky for us, someone made a PDF file that goes extrememly in-depth about how Vector works and the boot process, and even the OSKR documentation goes a little bit into detail.

So, Vector is based on Linux running on top of an Android based bootloader.

Oh also some people may be wondering why you have to be sent a custom OSKR firmware instead of just using a regular default one, and the reason why is because Vector is extremely secure.

So, according to the OSKR documentation, when you boot Vector, 3 things happen:

  • A cryptographic code stored in the Qualcom TrustZone, a special part of the CPU for sensitive info that is (by default) read-only and protected by e-fuses, is verified against the cryptographic code in the ABOOT (?) partition. If this fails, Vector will not boot.
  • If that passes, the ABOOT partition uses a different security key inside of it to verify the main BOOT partition. Again, if this fails, Vector will not boot. According to DDL, "The BOOT partition is a skeleton linux operating system to bootstrap the main system."
  • If that passes, the BOOT partition loads the SYSTEM partition. Inside the SYSTEM partition is an SHA256 hash of the manifest.json file (manifest.sha256). The manifest.json file constains important info (like telling the robot if its a developer robot or not). A hash is a string of characters generated from the files properties/contents. If the manifest.json file is edited, the manifest.sha256 file will be DIFFERENT than the actual manifest.json file, and Vector will not boot.

The OSKR firmware is pretty much identical to the regular firmware, except on a normal Vector, contained in the manifest.json file is a line that says something along the lines of "DEV_VECTOR = 0", where as on an OSKR Vector, it says "DEV_VECTOR = 1" (in computers, 0 means off/no and 1 means on/yes)

I think you can see our issue now.

If we change the manifest.json file, the manifest.sha256 file will be different to the hash of the actual manifest.json. Oh, but couldn't we just edit the manifest.sha256 file to actually match up? No. This is because Vector stores a permenant copy of the manifest.sha256 file in Qualcomm's TrustZone. Wouldn't this cause an issue because the manifest.json file is constantly being updated, meaning the manifest.json file wouldn't match up? No, because the manifest.json file is basically never changed.

So, what the OSKR firmware does, it it first gets verification from Qualcomm's TrustZone using your ESN/QSN to edit the contents, changing the hash to the correct one. Then, it simply erases literally the entire robot's storage, then installs OSKR firmware.

The issue here is that it erases the entire robot's storage, including the recovery partition. This means that if anything AT ALL goes wrong during the installation, you're basically screwed because there is no more recovery partition to install software from.

But that's besides the point

THAT is why you need a custom OSKR firmware for OSKR to work properly. If you want more info/feel i havent explained it very well, go to the OSKR documentation.
